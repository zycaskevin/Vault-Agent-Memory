# Vault-for-LLM v0.6.61

This is a remote-reader compatibility release.

## What's Fixed

- `vault remote map` and `vault remote read` now accept the IDs returned by Supabase search directly, whether they are local integer IDs or Supabase UUIDs.
- MCP tools `vault_remote_map_show` and `vault_remote_read_range` now accept the same integer-or-UUID remote IDs.
- CLI remote map/read now support the same agent policy flags as remote search: `--agent-id`, `--include-private`, and `--max-sensitivity`.
- `docs/supabase_read_policy.sql` now uses a text RPC boundary for remote IDs, so the same guarded reader functions work with integer or UUID schemas.

## Why It Matters

Hosted agents should be able to follow the normal safe path:

```text
vault_remote_search -> vault_remote_map_show -> vault_remote_read_range
```

Before this release, UUID-based Supabase deployments could search successfully but fail at map/read because the reader expected an integer ID.

If you already deployed the remote reader SQL, reapply `docs/supabase_read_policy.sql` after upgrading so Supabase replaces the old integer-only RPC signatures.

## Safety Boundary

Malformed strings are still rejected. Remote reads still go through the guarded Supabase RPCs and agent policy parameters.
