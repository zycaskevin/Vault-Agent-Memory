# Vault Agent Memory v0.8.0

v0.8.0 promotes the Central Memory Station foundation from release gate to
daily-use runtime for agent memory.

Release maturity: **Public Beta / Developer Preview**. The local-first Vault is
the source of truth. Supabase, Gateway, and central vector search are advanced
read/candidate-sync surfaces, not multi-master active memory.

## What changed

- Added the daily-loop runtime status surface for scheduled automation,
  candidate queue health, human-review cards, and remote sync health.
- Added a pure report refresh path so operators can rebuild the latest
  daily-loop report without creating new candidates.
- Hardened Supabase Document Map sync by translating local SQLite knowledge IDs
  to remote Supabase UUIDs before node and claim upserts.
- Treated dry-run sync reports as previews rather than completed remote syncs.
- Added trusted-host service-role sync support for controlled local operators.

## Install

```bash
pip install --upgrade "vault-for-llm[mcp,supabase]==0.8.0"
```

## First check

```bash
vault daily-loop status --json
vault remote status --json
vault daily-loop report --refresh --write-report
```

## Known limitations

- Remote or hosted agents should use anon/scoped credentials for approved reads
  and candidate submission only; service-role keys belong on the trusted sync
  host.
- Central vector search indexes reviewed safe summaries/previews as a derived
  read layer. It does not make Supabase the active-memory source of truth.
- Semantic query text may be sent to the configured embedding provider when
  remote semantic search is enabled.
- Automation remains candidate-first and report-first. Governed-auto can close
  routine low-risk metadata/dedup review noise, but freshness, convergence,
  consolidation, sensitive, and strategic work remains visible for review.
- LoCoMo / LongMemEval claims are limited to retrieval-only source-hit probes
  unless a comparable final-QA answerer and judge run is published.
