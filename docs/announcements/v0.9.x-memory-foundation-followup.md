# Vault Agent Memory v0.9.x memory foundation follow-up

This is the release-note draft for the main-branch work after the `v0.9.0`
Public Beta / Developer Preview tag.

The goal is to make the product boundary easier to understand:

- Vault still works standalone through CLI, MCP, Gateway, and local SQLite.
- Vault can also act as the governed memory foundation underneath other agents
  and memory frameworks.
- SQLite, self-host central hosts, Supabase, Postgres-like future adapters, and
  Vault Cloud are deployment backends, not different product meanings.

## What changed on main

- Added backend-agnostic positioning around the Vault Governance Contract and
  Memory Provider Interface.
- Added deployment-mode guidance for Local Vault, Trusted Local Central Memory
  Host, Supabase Adapter, and future Vault Cloud.
- Added the Self-host Central Memory Host specification and operator runbook.
- Added the draft Vault Memory API specification for create, search, read,
  update, delete, audit, timeline, and future sync/link/promote surfaces.
- Added an additive Gateway facade for the first Vault Memory API paths.
- Added the first runtime Memory Provider Interface and default SQLite provider
  facade, surfaced through Gateway health and OpenAPI.
- Started provider-read adoption for `/memory/search` and `/memory/{id}` as a
  shadow metadata probe while keeping the legacy Gateway policy gate as the
  authoritative result filter. Search probes inspect only ids already returned
  by that policy gate.
- Added read-policy filtering to the default SQLite provider for `agent_id`,
  `include_private`, and `max_sensitivity`.
- Added an opt-in provider-backed `/memory/search` result adapter through
  `result_adapter=provider`. It is preview-only and compact-result-only; the
  default `/memory/search` authority remains the legacy Gateway
  policy-filtered path until provider-backed parity is proven.
- Added an opt-in provider-backed `/memory/{id}` bounded-read adapter through
  `result_adapter=provider`. It uses provider read-policy filtering before
  returning a requested line range.
- Added a metadata-only provider adapter parity report helper. It compares
  legacy Gateway search/read behavior against provider-backed preview adapters
  and is the promotion gate before any future default authority switch.
- Added `vault memory-api parity-report` so operators can run that gate from
  the CLI without changing default Gateway or provider authority.

## Safety boundary

- The facade is candidate-first: remote or untrusted create/update/delete
  requests submit review candidates instead of writing or deleting active
  memory directly.
- DELETE is not a hard delete in the current facade. It submits a soft-delete
  review candidate.
- Existing CLI, MCP tools, local SQLite usage, and legacy Gateway paths remain
  supported.
- The SQLite provider facade preserves candidate-first remote writes, trusted
  local active-memory updates, metadata-only audit/timeline surfaces, and no
  hard delete by remote agents.
- Provider-read adoption does not return provider raw rows or raw content; it
  only reports metadata about the active provider path and avoids hidden-result
  count side channels.
- Provider read filtering is enabled before provider-backed Memory API results
  become authoritative.
- The provider-backed search adapter is opt-in and returns only compact,
  policy-filtered result rows. It does not expose provider raw rows or raw
  memory content.
- The provider-backed read adapter is opt-in and bounded. It returns the
  requested line range after provider policy filtering, not the full provider
  raw row or full memory body.
- The parity report is report-only. It compares search ids, read access
  decisions, and bounded read content hashes without returning raw memory
  content, raw query text, or changing the default Gateway authority.
- The CLI parity command requires explicit probes (`--search-query` or
  `--read-range`) so an empty report cannot be mistaken for a passing gate.
- Semantic index providers such as Qdrant are index backends only, not the
  default source-of-truth memory store.

## How to describe it publicly

Recommended:

```text
Vault is a local-first, backend-agnostic memory governance layer for AI agents.
It works standalone, and it can also provide the governed memory foundation for
other agent and memory frameworks.
```

Avoid:

```text
Vault is a multi-master cloud memory database.
Vault requires Supabase, Letta, mem0, or Vault Cloud.
The Memory API allows remote agents to mutate active memory directly.
```

## Release status

These changes are on `main` after the `v0.9.0` tag. Treat this as a `0.9.x`
candidate note until a new version is tagged, built, and published.
