# 升級指南：v0.8.0 → v0.10.0

**升級風險**：低  
**向後相容性**：✅ 完全向後相容，無破壞性變更  
**預估升級時間**：5-15 分鐘（取決於記憶庫大小）

---

## 升級前檢查清單

在開始升級之前，請確認以下事項：

- [ ] 已備份當前 vault.db：`vault db backup --verify`
- [ ] 已備份設定檔與自訂腳本
- [ ] 確認 Python 版本 >= 3.10（v0.10.0 支援 3.10-3.13）
- [ ] 確認有穩定的網路連線（用於 pip 安裝與 embedding API）
- [ ] 若使用 Supabase 同步，已備份 Supabase 專案設定

---

## 升級步驟

### 步驟 1：升級套件

```bash
# 升級到最新版本
pip install --upgrade "vault-for-llm[mcp]==0.10.0"

# 驗證版本
vault --version
# 預期輸出：vault-for-llm 0.10.0
```

> **注意**：國內 PyPI 鏡像源可能延後同步，若無法找到 0.10.0 版本，請使用官方源：
> ```bash
> pip install --upgrade "vault-for-llm[mcp]==0.10.0" -i https://pypi.org/simple/
> ```

### 步驟 2：自動資料庫遷移

第一次執行任何 vault 指令時，系統會自動執行 schema 遷移。

```bash
# 進入專案目錄並檢查狀態（會自動觸發遷移）
cd /path/to/your/vault-project
vault db status

# 確認遷移成功
# 預期看到 schema_version: 14（或更高）
```

**遷移內容**（schema v12 → v14）：
- 新增向量索引相關表格
- 新增治理合約相關欄位
- 新增記憶提供者介面所需的中繼資料
- 最佳化部分索引效能

### 步驟 3：驗證核心功能

```bash
# 健康檢查
vault daily-loop status

# 安全檢查
vault security doctor

# 搜尋測試（確認既有記憶可讀取）
vault search "測試關鍵字" --limit 3
```

### 步驟 4：（選擇性）啟用新功能

#### 4a. 向量索引

v0.10.0 新增中央衍生向量索引（Central Derived Vector Index），這是一個**可重建的搜尋加速層**，不是第二個記憶庫。

```bash
# 檢查向量索引狀態
vault vector-index status

# 初次建構（乾跑，先看計畫）
vault vector-index repair --full

# 確認無誤後實際執行
# 小型記憶庫（< 100 條）：
vault vector-index repair --full --apply

# 大型記憶庫（> 500 條）：建議使用批次模式
vault vector-index repair --apply --batch-size 50
```

#### 4b. Memory API（Gateway）

v0.10.0 新增完整的 HTTP Memory API（`/memory/*` 端點），適用於不支援 MCP 的整合場景。

```bash
# 檢查 Gateway 狀態
vault gateway status --profile memory_api

# 查看 OpenAPI 規格
# 啟動 Gateway 後造訪 /openapi.json
```

#### 4c. 記憶提供者介面（進階）

若你需要自訂後端或擴充記憶來源，v0.10.0 新增了標準化的 Memory Provider Interface。

```bash
# 奇偶校驗：比較 legacy 路徑與 provider 路徑的結果一致性
vault memory-api parity-report
```

## 設定變更說明

### 新增設定項目
| 設定 | 預設值 | 說明 |
|------|--------|------|
| `VAULT_VECTOR_INDEX_ENABLED` | auto | 向量索引開關 |
| `VAULT_GATEWAY_RATE_LIMIT` | 60 | Gateway 每分鐘速率限制 |
| `VAULT_GATEWAY_AUTH_FAILURE_LIMIT` | 10 | 認證失敗鎖定門檻 |
| `VAULT_GATEWAY_AUTH_LOCKOUT_SECONDS` | 300 | 認證鎖定時長（秒） |
| `VAULT_MEMORY_API_ENABLED` | false | Memory API 端點開關 |

### 不變更的設定
- 所有 v0.8.0 的設定在 v0.10.0 中繼續有效
- 自動化政策（automation_policy.yaml）格式不變
- Agent 註冊格式不變
- MCP 工具設定檔不變

## 常見問題

### Q: 升級後我的記憶會不見嗎？
不會。v0.10.0 完全向後相容，schema 遷移只會新增表格和欄位，不會修改或刪除既有資料。升級前務必備份。

### Q: 升級需要多久？
- 小型記憶庫（< 100 條）：5 分鐘內
- 中型記憶庫（100-1000 條）：10-15 分鐘
- 若需建構向量索引：視 embedding API 速度而定，每 100 條約 1-5 分鐘

### Q: 升級後可以降回 v0.8.0 嗎？
可以，但 schema 遷移後的資料庫無法直接給 v0.8.0 使用。若需降級，請使用升級前的備份還原。

### Q: 現有的 MCP 整合需要改動嗎？
不需要。所有 v0.8.0 的 MCP 工具在 v0.10.0 中保持不變，新工具為附加性質。

### Q: Supabase 同步需要重新設定嗎？
不需要。Supabase 同步的設定和行為與 v0.8.0 一致。若要使用向量同步功能，需要執行 `vault remote setup` 更新 schema。

### Q: 向量索引佔用多少空間？
大約是原始文本大小的 20-50%（取決於 embedding 維度）。預設 1536 維的情況下，1000 條記憶約佔 100-200MB。

## 回滾步驟

若升級後遇到問題，可按以下步驟回滾：

```bash
# 1. 降級套件
pip install "vault-for-llm[mcp]==0.8.0"

# 2. 還原資料庫備份
vault db restore --backup /path/to/backup-file.db

# 3. 驗證
vault --version
vault db status
```
