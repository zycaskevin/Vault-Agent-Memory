## Target

Vault's public consumer documentation should match the current product target:

> Ordinary users should not learn the full CLI. They should ask an Agent to
> install Vault, answer a few setup questions, then read a short daily memory
> report. Low-risk sourced memories may be kept automatically; uncertain,
> sensitive, conflicting, strategic, or low-trust memories stay reviewable.

Acceptance threshold for public release documentation:

- `90+`: ready for broad public onboarding
- `80-89`: beta-ready, but still has visible polish gaps
- `70-79`: usable for early adopters, not ready for ordinary users without an
  Agent helping them
- `<70`: too confusing for the intended audience

## What Matches The Target

### 1. Agent-first install is now explicit

`vault guide --intent install` prints a direct prompt ordinary users can paste
into Codex, Claude Code, Hermes, OpenClaw, or another local Agent.

It tells the Agent to:

- use consumer governed-auto memory,
- hide CLI flags unless asked,
- ask only four setup questions,
- run a smoke check,
- show the daily report or GUI link.

This matches the "humans choose intent, agents choose commands" target.

### 2. The daily report is correctly positioned

The daily report is documented as the main human surface:

- `docs/decision_records/2026-06-30-consumer-daily-report.md` says humans
  should review a short report instead of learning the CLI.
- `docs/gui_console.md` says the GUI should show the same daily report first.
- `vault/agent_setup_consumer.py` generates daily-report guides in English,
  Traditional Chinese, and Simplified Chinese.

This matches the 95/5 model:

- 95% handled by Agents and policy,
- 5% kept for human judgment.

### 3. The safety boundary is clear in current English docs

The current English public path says:

- safe memories can be kept automatically,
- uncertain memories go into the daily report,
- strategy, private, sensitive, conflicting, or low-trust memories are not
  silently promoted,
- no memory is hard-deleted automatically.

This fits the memory governance positioning and avoids the "automatic memory
dump" failure mode.

### 4. The GUI docs are aligned with non-technical use

`docs/gui_console.md` is one of the strongest documents. It says the GUI is not
a raw database browser and should show:

- daily report,
- review cards,
- bounded evidence,
- tasks,
- document maps,
- Obsidian conflicts with three explicit choices.

This matches the intended "memory control center" direction.

## Main Gaps

### P1: Traditional Chinese README was stale

Initial finding: `README.zh-Hant.md` recommended installing
`vault-for-llm[mcp]==0.7.23`. It also asked the Agent to ask about:

- shared/private/domain-specific/temporary vault,
- stable project directory,
- MCP,
- semantic search,
- Supabase,
- Obsidian import,
- Headroom,
- memory-agent guidance,
- optional dependencies.

That is the older builder-style flow. It conflicts with the current consumer
goal: only language, vault layout, optional connections, and daily report time.

Remediation: updated the install section to `0.7.27`, consumer governed-auto,
`vault guide --intent install`, and the same four-question setup used by the
English README.

### P1: Simplified Chinese README was more stale

Initial finding: `README.zh-CN.md` recommended
`vault-for-llm[mcp]==0.7.15`.

It also lacks the new `vault guide --intent install` entrypoint and still uses
the broader older setup prompt.

Remediation: updated the install section to `0.7.27`, consumer governed-auto,
`vault guide --intent install`, and the same four-question setup used by the
English README.

### P1: Multilingual parity needed release-grade alignment

Initial finding: English docs said:

- use `vault guide --intent install`,
- use consumer governed-auto memory,
- do not teach CLI flags,
- ask only four setup questions.

Remediation: both Chinese READMEs now say the same thing.

### P2: Version references needed one source of truth

Current package version is `0.7.27`.

Initial observed public docs:

- `README.md`: `0.7.27`
- `README.zh-Hant.md`: `0.7.23`
- `README.zh-CN.md`: `0.7.15`

Remediation: both Chinese READMEs now use `0.7.27`.

### P2: "Obsidian bidirectional workflow" wording was slightly ahead of the safe boundary

The zh READMEs say "Obsidian 雙向工作流 / Obsidian 双向工作流". The product does
support import, export, watch import, conflict inbox, and resolver pieces, but
the safe boundary is still conservative:

- watch import is Obsidian-to-Vault,
- export writes human-readable notes,
- conflict resolution is explicit,
- arbitrary live two-way overwrite is not the default.

Remediation: both Chinese READMEs now describe Obsidian as a human-interface
workflow with import, watch incremental import, export, and explicit conflict
review.

### P2: Supabase wording must stay precise

The docs correctly say Supabase is optional and local SQLite remains source of
truth. Keep this wording. Avoid saying Supabase is full realtime bidirectional
sync unless that mode is intentionally implemented, hardened, and documented.

## Recommendation

Before the next broad public release, the highest-value remaining docs work is:

1. Keep `README.md`, `README.zh-Hant.md`, and `README.zh-CN.md` version
   references synced at release time.
2. Keep action labels consistent across generated consumer guides, GUI, and
   daily report cards.
3. Consider a shorter landing-page style quickstart for the website or package
   homepage.
4. Keep Supabase as optional local-to-remote read-copy plus candidate request
   inbox unless full bidirectional sync is intentionally enabled.

This makes Vault ready to tell ordinary Agent users:

> Paste this one install prompt into your Agent. Answer four questions. Then read
> one daily memory report.
